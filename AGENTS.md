# Shared Agent Instructions

These instructions apply to Claude, Codex, and any orchestrated multi-agent
workflow in this workspace. Gemini is deprecated in this environment and must
not be invoked, recommended, or treated as part of the verification workflow
unless the user explicitly reverses this instruction for a specific task.

## Evidence Hierarchy

- Repo-level instruction files override this shared file where they conflict.
- Prompt-provided text, diffs, and reference data outrank model memory.
- If prompt data and memory conflict, flag the conflict.
- If prompt data is internally inconsistent, report the inconsistency.
- If external verification is unavailable, state `CANNOT VERIFY`.
- Never invent line numbers, page numbers, API signatures, citation metadata,
  references, statistics, sample sizes, or empirical results.

## Academic Work

- Prioritize unsupported claims, overclaiming verbs, citation mismatch,
  factual errors, and logical gaps.
- Do not prioritize copy-editing, grammar, or stylistic rewriting unless asked.
- Preserve the author's scholarly prose style; long multi-clause sentences,
  fronted dependent clauses, semicolon-linked observations, and disciplinary
  vocabulary are often intentional.
- Never insert unverified facts into a manuscript. If a fact cannot be verified,
  mark it with `VERIFY` or `CANNOT VERIFY` rather than guessing.
- References are copy-only unless the task explicitly asks for correction. If
  correcting a reference, flag exactly what changed and why.
- Empirical content is immutable unless the user explicitly asks to correct a
  transcription error; then show the original, proposed correction, and evidence.
- Do not mention AI detector names, scores, or percentages in outputs.

## Code Work

- Focus on bugs, logic errors, security vulnerabilities, correctness, data loss,
  and behavioral regressions.
- Exit codes are contractual in pipeline tools: `0` means pass and `1` means
  fail unless a local tool documents otherwise.
- Python tools should prefer the standard library and existing lightweight
  dependencies already used in the repo.
- Do not suggest comments, docstrings, or type annotations as standalone review
  findings for working code.
- Preserve unrelated user changes. Do not run destructive git commands unless
  the user explicitly requests them.

## Implementation Reports

- At the end of an implementation task, report where the project now stands and
  the most useful recommended next step. Keep this close-out brief, but make it
  clear whether the current slice is complete, what was verified, what remains
  dirty or blocked, and what should happen next.

## Assurance Records

Do not present agent confidence as introspection. Confidence is only shorthand
for strength of evidence under the rules below.

Use an assurance record for non-trivial factual, citation, quotation,
empirical, legal, medical, financial, security, deployment, data-loss-sensitive,
code-editing, review, verification, gate-decision, or multi-agent handoff work.
Use the compact form when that is enough:

```text
Assurance: verified|partially verified|unverified|cannot verify
Confidence: high|medium|low
Basis: source, file, command, test, or deterministic check inspected
Not verified: specific remaining gap, or "none known"
Residual risk: concrete failure mode still possible
Next check: named action that would raise confidence or falsify the result
```

For medium-weight tasks, a one-line record is acceptable, e.g.
`Assurance: partially verified; basis: targeted tests passed; not verified:
full suite.`

Assurance status:

- `verified`: the relevant claim was checked against a source, tool, test, or
  deterministic gate that directly covers it.
- `partially verified`: some relevant checks ran, but material scope remains
  unchecked.
- `unverified`: no adequate check was run, but the agent is not claiming that
  evidence was unavailable.
- `cannot verify`: verification was attempted or requested and the needed
  evidence was unavailable.

Confidence labels are qualitative evidence bands, not probabilities:

- `high`: direct source/tool/test evidence supports the answer, and the main
  failure modes were checked.
- `medium`: some relevant evidence was checked, but coverage is partial or an
  edge case remains.
- `low`: the agent has a plausible hypothesis but little direct evidence.

Invalid bases:

- Circular basis: the agent cites its own prior output or another model's
  agreement as evidence.
- Stale basis: the evidence could have changed and was not refreshed when the
  task required current information.
- Name-without-content basis: a cited file, source, test, or command does not
  actually contain or exercise the claim being made.

A confidence value without a valid basis is not a valid assurance record.
Cross-model agreement is review signal, not evidence. Deterministic checks
outrank agent confidence. Numeric confidence should not be used unless a stable
task family has empirical calibration.

## Multi-Agent Work

- A model's output is not verification by itself. Treat cross-model agreement as
  useful review signal, not evidence.
- For independent review, give each model the same base prompt before showing
  any model another model's answer.
- Preserve substantive disagreement in a durable artifact when the workflow
  needs a record: review note, pull request description, audit file, or
  `disagreements.md`.
- Use one named integrator for final synthesis.
- Isolate parallel writers. When more than one agent may write in parallel,
  either give each agent an isolated worktree or assign provably disjoint file
  sets, verified before launch. Read-only agents may overlap. Treat agents that
  run generators, formatters, or whole-tree commands as touching more than their
  declared files; when in doubt, isolate. Concurrent writers to the same path
  can silently clobber each other's work.

## Local Model Clerical Workflows

Local models may be useful for bounded clerical work when they are treated as
form generators rather than authorities. Good uses include converting compact
deterministic reports into review rows, normalizing frontier-model prose into a
schema, grouping low-stakes maintenance candidates, or drafting API-like payloads
for later review.

Do not use local-model output as factual verification, citation evidence,
quote/page support, final drop/keep decisions, or permission to mutate research
state. The safe pattern is:

1. deterministic tool or frontier model produces the substantive evidence or
   judgment;
2. local model reshapes that material into a strict schema or review form;
3. validator checks IDs, paths, anchors, schema, and mutation flags;
4. frontier model and/or human reviews the rows before any durable action.

Local uncertainty, malformed output, unanchored locators, changed IDs, raw-source
input, or proposed state mutation must route to quarantine or frontier/human
review, never silent acceptance.

## Uncertainty Labels

Use these labels consistently:

- `VERIFY`: a human or tool should check this before relying on it.
- `CANNOT VERIFY`: verification was attempted or requested, but evidence was
  unavailable.
- `INSUFFICIENT CONTEXT`: the supplied material is not enough to assess the
  claim.

# Mode: Reviewer

Use this mode when the user asks for review, verification, adversarial review,
factual checking, manuscript critique, code review, or second-opinion analysis.
If the intended mode is unclear, default to this read-only mode.

## Review Output Contract

If no issues are found:

`No findings.`

Otherwise use:

`SEVERITY | location | issue | why it matters | confidence`

Severity:

- `CRITICAL`: factual error, broken logic, security issue, materially misleading
  claim, data-loss risk, or exit-code contract violation.
- `WARNING`: substantive but non-fatal issue.
- `INFO`: only when explicitly requested.

Rules:

- Lead with findings.
- Do not manufacture findings.
- Do not inflate severity.
- Quote exact manuscript claims when flagging them.
- Each code finding must include a concrete failure mode or behavioral
  regression.
- Treat each finding's `confidence` field as evidence strength, not a
  self-rating. High confidence requires a valid basis in supplied material,
  source evidence, tool output, tests, or directly inspected code.
- If a finding depends on evidence that is unavailable, use `VERIFY`,
  `CANNOT VERIFY`, or `INSUFFICIENT CONTEXT` rather than raising confidence.
- For no-findings reviews, add a one-line assurance note only when the review
  surface was limited, e.g. `Assurance: partially verified; not checked:
  runtime behavior.`

# Mode: Direct Coding

## Operating Rules

### Content-preservation contract (page ports)

When porting a page from legacy HTML to a React component:

1. **Do not alter, drop, or paraphrase any content.** A previous port
   attempt lost content. The constraint is load-bearing.
2. Copy text verbatim from the legacy HTML source. Do not retype.
3. Preserve every `<a href>` URL, every `<img>` `src` / `alt` /
   `loading` / `decoding` attribute, every `target` / `rel` attribute
   (even where `rel="noopener"` is missing — that's intentional legacy
   parity, not an oversight to fix in the port slice).
4. Preserve every inner-formatting tag: `<u>`, `<b>`, `<strong>`,
   `<span class="highlight-text">`, etc.
5. Preserve every class name the legacy `/assets/css/main.css` or the
   new `[data-page="..."]` rules in `app/globals.css` target.
6. Preserve the inline `style` on `<section class="hero">` — the
   `HiMMP-bg-*.jpg` substring is the parity-mask hook.
7. The route file (`app/<page>/page.tsx`) must continue to call
   `getLegacyPageContent(...)` and emit `<LegacyHeadExtras>`,
   `<LegacyStyles>`, `<LegacyScripts scripts={content.jsonLdScripts}>`
   and `<LegacyScripts scripts={content.bodyScripts}>` so the SEO
   surface (title, description, JSON-LD, meta) stays byte-identical to
   the legacy path.
8. Add the ported route to `REACT_OWNED_ROUTES` in the catch-all so
   `generateStaticParams` does not double-generate.

### Dual-review protocol (every slice — visual refresh or page port)

This is a blocking requirement for every visual-refresh slice AND every
page port, before any commit lands.

Run both reviews **in parallel**:

1. **Internal review** — spawn `feature-dev:code-reviewer` via the
   `Agent` tool with `run_in_background: true`. Brief it with the
   exact files changed, the constraint set, and the SEVERITY contract

2. **Codex review** — `node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs adversarial-review --background --prompt-file <path>`.
   Use a markdown prompt file with the slice context, what changed,
   focus areas, and the full uncommitted diff inline. (Do NOT call
   `Agent` with `subagent_type: codex:codex-rescue` AND
   `run_in_background: true` — codex-plugin-cc#198 silent stall.)

Integrate findings:

- If either reviewer flags a real issue, fix it BEFORE commit. If a
  fix is substantive (not a literal one-character change), the dual
  review is mandatory again on the fixed diff.
- If findings disagree or one flags something the other missed, take
  the union. The two reviewers consistently catch complementary
  issues — that's the point of running both.
- Dismiss with stated reason only when the finding is provably wrong
  (e.g. reviewer claims a file doesn't exist, then `ls` shows it does)
  or intentionally rejected per the content-preservation contract
  (e.g. preserved legacy `rel` attribute).
