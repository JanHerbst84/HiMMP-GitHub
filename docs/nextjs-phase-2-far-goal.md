## What this document is

A north-star statement for the HiMMP Next.js Phase 2 work. It names the **end-state** the project is moving toward, the **acceptance criteria** that define "done", what is **explicitly excluded**, and the **operational shape** of getting there. ## End-state

The HiMMP website at `https://himmp.net` is a fully React-owned, token-driven static export served from the current production host (a Hostinger VPS at `5.182.18.217`, in active service since 2026-05-12). The deployment authority is `~/.claude/vps-infrastructure.md` per the global workspace instruction "Before any VPS / ports / deployment changes: read `~/.claude/vps-infrastructure.md` (authoritative)"; `docs/hostinger-deployment.md` is a project-local subordinate record that captures the HiMMP-specific deploy procedure (release-dir layout, nginx site config path, PHP-FPM socket, smoke commands) and must stay consistent with the global registry. Both records confirm the HiMMP entry; if they ever diverge, the global registry wins and the project-local doc is updated. Every page — the 12 root routes and the 15 findings chapters — is rendered from a hand-written or auto-generated React component. The legacy `/assets/css/main.css` and `/assets/css/responsive.css` are no longer loaded by the browser (no `<link rel="stylesheet">` references either file). The two legacy CSS files remain on disk pending the D-7 gate (see "Explicitly out of scope" below). All component styling flows through `app/tokens.css` (palette, type scale, spacing, motion) and `app/globals.css` (component rules) plus a small number of colocated CSS modules for genuinely scoped chrome.

The dark-academic palette is consistent across **the surfaces controlled by `app/globals.css`, `app/tokens.css`, and any colocated `*.module.css`**. Bone is the page bg; graphite is text and dark chrome; mint is a hairline accent; sulfur is reserved for active audio states; paper (off-bone) is the card/panel surface. Within that scope, there is no rule that declares a literal colour value not in the whitelist (criterion 3 audit).

**This claim does not extend to React-owned page components** (`src/site/components/pages/**/*.tsx`, including the 15 auto-generated findings chapter components under `src/site/components/pages/findings/*.tsx`, plus any inline `style` expressions in `app/**/*.tsx` route wrappers). Those components preserve every legacy `style` attribute verbatim per the content-preservation contract (`CLAUDE.md` §Phase 2), and the inline literals visible in them (`HomeMain.tsx` shipping `#fff` / `#333` / gradients; `PublicationsPage.tsx` shipping `#f0f8f5` / `#5DC69F` / `#666`; the 15 findings chapter components shipping similar literals; similar in other page components) are intentional carry-overs from the legacy HTML. The far-goal **does not** claim a fully-token-driven HiMMP site: it claims that the **legacy stylesheet links are retired** while **inline legacy styles in page components remain tracked exceptions** under the audit snapshot in Acceptance Criterion 4. Modernising those inline styles is post-far-goal work (provisionally D-10), gated on a revision to the content-preservation contract that lets a port slice substitute literal hex for token references on a per-element basis.

The contact form continues to flow through `/contact-handler.php` + `/get-csrf-token.php` via PHP-FPM, untouched by the React migration. Audio + video assets serve directly from `assets/audio/` and YouTube embeds respectively, both deliberately preserved.

## Operational shape

The far goal is reached by completing D-1 (9 slices, sequenced J → L → G → F → I → H → D → responsive → link retirement gate). **D-8 (within-chapter TOC React component) lands as a Family L sub-slice during the Family L slice, per the deferred-scope record D-8 entry** — it is in scope and a prerequisite for criterion 9. D-3, D-6, D-7, D-10 are separate post-D-1 proposals. They are NOT bundled into D-1. If during a D-1 slice the team realises a deferred item's gating constraint has incidentally been resolved, the deferred item is **not** re-opened in the same slice — re-open it as a separate proposal that cites the gate as resolved (per `docs/nextjs-phase-2-design-refresh-future.md` §"Re-Entry Checklist For A Future Agent").

Per-slice protocol stays as established in `CLAUDE.md` §"Dual-review protocol (every slice — visual refresh or page port)": token-driven CSS, content-preservation contract, no markup edits, no controller-binding renames. The Slice loop is the working unit and follows the controlling protocol verbatim:

1. **Do the slice** — write the changes against the constraint set.
2. **Run both reviews in parallel on the same diff** — `feature-dev:code-reviewer` spawned via the `Agent` tool with `run_in_background: true`, AND `node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs adversarial-review --background --prompt-file <path>`. The Codex dispatch is canonically the background + prompt-file path for **all** risk classes — Low, Medium, and High. The interactive `/codex:adversarial-review` slash command is NOT an acceptable substitute because it can be launched without the same prompt-file context and without preserving the simultaneous-dispatch property. Both dispatches happen simultaneously to preserve the independent-review property; sequential review is a protocol violation.
3. **Integrate findings** — take the union of both reviewers' findings; fix substantive issues before commit.
4. **If the fix is more than a literal one-character change** — rerun both reviews in parallel on the fixed diff. Do not treat a post-fix diff as cleared without re-running both.
5. **Commit** — Claude (or a human) performs the commit locally with the dual-review template from `CLAUDE.md` in the commit message body. Codex does not commit autonomously.
6. **Push** — on the Linux work machine, dispatch `~/.claude/scripts/codex-push.sh <repo-path>` which uses Codex as the push transport to perform `git push origin main` (the harness classifier blocks direct push from Claude; this is the authorised routing per the global CLAUDE.md). Codex is the push transport, not the integrator — the commit content is settled before dispatch.

The slice plan defines risk classes per family (Low → High); high-risk slices (4.6 audio chrome, 4.7 findings reader, 4.9 link retirement) additionally require human confirmation before the push step.

Realistic calendar: 2–3 working weeks at one slice per session for D-1 alone (9 slices including D-8 as a Family L sub-slice, plus the audit scripts named in criteria 3, 4, 6 as scaffolding for the gates). The seven-day stability observation in acceptance criterion 10 adds a final week. D-3 / D-6 / D-7 / D-10 are separate work units on top, each with their own calendar that this far-goal does not commit to.

**Escalation path on slice-review failure.** If a D-1 slice fails dual review (CRITICAL findings from either reviewer) three consecutive attempts on the same diff, the slice is paused and the failure is escalated to JPH as a design-decision question. The far-goal does not constrain the resolution of that conversation — the slice may be re-scoped, deferred to a later workstream, or dropped from the far-goal acceptance criteria with explicit doc update.

## Why this is the right far goal

It is bounded. The work is enumerated, not open-ended. Every slice has acceptance criteria. The "done" condition is mechanically checkable (audit scripts + Playwright + grep + Hostinger logs + Matomo).

It is reversible at every checkpoint. Each of the 9 D-1 slices commits independently and reverts cleanly. The final link-retirement slice is two lines of code to revert if a regression surfaces.

It does not gamble on new infrastructure or change controlling contracts. View Transitions, dark mode, content-architecture refactor, and PHP-to-API-route are all explicitly excluded. No new dependencies, no new build steps, no new hosting decisions. The content-preservation contract is preserved; modernising the inline TSX styles is a separate workstream gated on a contract revision.

It is the natural end of the Phase 2 thread. The Coexist v2 slice (shipped) put the dark-academic palette in place; D-1 absorbs the structural CSS, restores the within-chapter TOC affordance (D-8 inside Family L), and lets the legacy stylesheets be unloaded; the far-goal stops there. D-3, D-6, D-7, D-10 are Phase 3 conversations.
