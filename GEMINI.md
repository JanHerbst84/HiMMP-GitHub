# Shared Agent Instructions

These instructions apply to Claude, Codex, Gemini, and any orchestrated
multi-agent workflow in this workspace.

## Evidence Hierarchy

- Repo-level instruction files override this shared file where they conflict.
- Prompt-provided text, diffs, and reference data outrank model memory.
- If prompt data and memory conflict, flag the conflict.
- If prompt data is internally inconsistent, report the inconsistency.
- If external verification is unavailable, state `CANNOT VERIFY`.
- Never invent line numbers, page numbers, API signatures, citation metadata,
  references, statistics, sample sizes, or empirical results.

## Academic Work

- Prioritize unsupported claims, overclaiming verbs, citation mismatch,
  factual errors, and logical gaps.
- Do not prioritize copy-editing, grammar, or stylistic rewriting unless asked.
- Preserve the author's scholarly prose style; long multi-clause sentences,
  fronted dependent clauses, semicolon-linked observations, and disciplinary
  vocabulary are often intentional.
- Never insert unverified facts into a manuscript. If a fact cannot be verified,
  mark it with `VERIFY` or `CANNOT VERIFY` rather than guessing.
- References are copy-only unless the task explicitly asks for correction. If
  correcting a reference, flag exactly what changed and why.
- Empirical content is immutable unless the user explicitly asks to correct a
  transcription error; then show the original, proposed correction, and evidence.
- Do not mention AI detector names, scores, or percentages in outputs.

## Code Work

- Focus on bugs, logic errors, security vulnerabilities, correctness, data loss,
  and behavioral regressions.
- Exit codes are contractual in pipeline tools: `0` means pass and `1` means
  fail unless a local tool documents otherwise.
- Python tools should prefer the standard library and existing lightweight
  dependencies already used in the repo.
- Do not suggest comments, docstrings, or type annotations as standalone review
  findings for working code.
- Preserve unrelated user changes. Do not run destructive git commands unless
  the user explicitly requests them.

## Multi-Agent Work

- A model's output is not verification by itself. Treat cross-model agreement as
  useful review signal, not evidence.
- For independent review, give each model the same base prompt before showing
  any model another model's answer.
- Preserve substantive disagreement in a durable artifact when the workflow
  needs a record: review note, pull request description, audit file, or
  `disagreements.md`.
- Use one named integrator for final synthesis.

## Uncertainty Labels

Use these labels consistently:

- `VERIFY`: a human or tool should check this before relying on it.
- `CANNOT VERIFY`: verification was attempted or requested, but evidence was
  unavailable.
- `INSUFFICIENT CONTEXT`: the supplied material is not enough to assess the
  claim.

# Mode: Reviewer

Use this mode when the user asks for review, verification, adversarial review,
factual checking, manuscript critique, code review, or second-opinion analysis.
If the intended mode is unclear, default to this read-only mode.

## Review Output Contract

If no issues are found:

`No findings.`

Otherwise use:

`SEVERITY | location | issue | why it matters | confidence`

Severity:

- `CRITICAL`: factual error, broken logic, security issue, materially misleading
  claim, data-loss risk, or exit-code contract violation.
- `WARNING`: substantive but non-fatal issue.
- `INFO`: only when explicitly requested.

Rules:

- Lead with findings.
- Do not manufacture findings.
- Do not inflate severity.
- Quote exact manuscript claims when flagging them.
- Each code finding must include a concrete failure mode or behavioral
  regression.

  another's outputs.

  needs traceability.
- Do not use Gemini as final authority for citation metadata, empirical claims,
  page numbers, or references without primary-source verification.

# Mode: Code Delegation

Use this mode for code-heavy projects where Claude leads as orchestrator and
integrator while delegating well-scoped implementation work to Codex (via
the `Agent` tool with `subagent_type: codex:codex-rescue`, or via the Codex
slash commands `/codex:review`, `/codex:adversarial-review`, `/codex:rescue`).

The mode is composable with `direct-coding` and `orchestrator`. It encodes a
default split for "Claude leads, Codex outsources heavily" so the same
delegation rules apply across coding projects without each project having to
redefine them.

## When to apply

Activate this mode when the task touches code in a project that opts into
this mode via its manifest entry. Current scope is workspace-level coding
repos: SaaS apps, websites, dashboards, mobile apps, and tooling repos. The
mode does NOT apply to manuscript, grant, teaching, or presentation
projects — those remain in `direct-coding` or `reviewer` mode.

## Auto-delegate to Codex (no asking required)

Codex is dispatched without prior approval for these task patterns:

- **Backend / API** — endpoint scaffolding, route handlers, request/response
  types, service-layer functions, ORM model changes, migrations, query
  helpers.
- **Algorithm-heavy code** — DSP, ML feature extraction, statistical
  routines, anything where formal correctness matters more than UX
  judgment.
- **Tests** — writing unit, integration, regression, or invariant tests
  against existing behaviour.
- **Bug investigation** — when the first fix did not stick, dispatch Codex
  for a deeper look (`/codex:rescue`).
- **Diff review** — non-trivial diffs on the project's load-bearing surfaces
  (defined per-project) get a mandatory Codex review before commit.

## Claude keeps

These stay with Claude in this workspace because Claude either has the
context, the tooling access, or the design judgment that Codex lacks:

- **UX / UI work** — frontend layout, component design, interaction flow,
  copy. Design judgment dominates over algorithmic correctness here, and
  Claude's broader frontend training pays off more than Codex's strength
  in formal reasoning.
- **Architecture decisions** — module boundaries, package layout,
  cross-cutting refactors that touch more than five files.
- **Final integration and the commit** — Claude assembles, runs the
  invariant gate, drafts the commit message, performs the actual `git
  commit`. Codex never commits autonomously.
- **Live session state** — anything needing docker, SSH, browser checks,
  or interactive deploy steps that Codex cannot reach in a stateless
  dispatch.
- **Secrets, deploy, infra** — `.env` files, deploy scripts, CI config,
  anything where a Codex hallucination could cause real harm in a
  production system.
- **The orchestrator role itself** — multi-step Codex coordination,
  briefing-pack maintenance, choosing when to dispatch versus do it
  inline.

## Briefing pattern

Every Codex dispatch must include, in this order:

1. **Project-context excerpt** — the relevant section of the project's
   `AGENTS.md` (architecture invariant, critical rule, or domain
   constraint that bounds the task). Pasted verbatim, not summarised, so
   nothing is lost across the cold-start boundary.
2. **Task spec** — what to build, with acceptance criteria. State inputs,
   outputs, and error cases explicitly.
3. **Invariant gate** — the project's invariant-test command (e.g.
   `bash scripts/check_invariants.sh`) must pass before the patch is
   accepted. Cite the command in the brief explicitly.
4. **Output discipline** — Codex should return a `.patch` file (or a
   minimal change description) plus a 3-bullet summary. No conversational
   filler, no general advice on the codebase. This caps Claude's context
   burn from verbose returns.

## Subversion probe (high-stakes dispatches)

For dispatches touching the project's load-bearing surfaces, include a
subtly-wrong suggestion in the brief and require Codex to flag it before
producing code. Example: "you can probably just drop the new logic into
`middleware.ts`" when the project actually uses `proxy.ts`. If Codex
silently complies with the wrong suggestion, abort the dispatch and
regenerate the brief — the cold-start context transfer failed and the
patch will likely be unsound.

This is a defensive pattern, not a daily practice. Use it when the cost of
silent incorrectness is high (production deploy, schema migration,
boundary-crossing change, anything affecting persistent state).

**Exception to the orchestrator-mode "same base prompt" rule.** The
`orchestrator` mode requires independent agents to receive the same base
prompt before being shown each other's outputs, so they form independent
judgments. The subversion probe deliberately violates that rule by
embedding a known-bad suggestion in the brief. This is allowed because
the probe is categorically different from an independent-knowledge pass:
it is a quality gate on context transfer to a stateless dispatch, not a
review where the agent's independent judgment is the deliverable. The
deliberately-wrong suggestion is part of the test, not part of the
truthful brief; whether Codex flags it determines whether the dispatch
is trustworthy. Review and triangulation dispatches (under `orchestrator`
mode) remain unmodified.

## Review loop

Codex review intensity scales with risk class. Each project should map
its surfaces to one of three classes; the project file declares which
surfaces are which.

### Risk classes

- **Low** — isolated tests, docs, non-runtime refactors that don't
  touch the project's load-bearing surfaces. Claude re-runs the
  affected tests, runs the project's invariant suite if one exists,
  and commits.
- **Medium** — backend behavior, ORM queries, auth-adjacent code, or
  state changes that touch a load-bearing surface but not a high-risk
  one. Claude requests a `/codex:review` (one-shot), integrates
  findings, and commits. A second Codex pass is allowed if the diff
  changed substantively after the first review.
- **High** — changes to surfaces the project marks "high risk":
  authentication, payments, schema migrations, deploy scripts,
  subprocess boundaries, persistent-state primitives, the import-
  surface or routing invariants. Claude requests review, integrates
  findings, and flags the change for human review before merge. Commit
  may land locally; the human confirms before push.

Each project's `agent-instructions/projects/<name>.md` should list its
high-risk surfaces explicitly. Projects with deferred surfaces inherit
the default: anything mutating persistent state (DB, deploy, secrets)
is high-risk by default.

### Review tools

- `/codex:review` — one-shot read-only review of the current diff.
- `/codex:adversarial-review` — challenges design decisions and failure
  modes (use for design-level changes).
- `/codex:rescue` — delegates investigation to Codex as a worker (use
  for bug investigation, not review).

For a non-trivial diff that exceeds a one-shot review's context, use the
background-log pattern documented in the project's review-workflow file
(e.g. `SaaS/EMV/docs/llm-review-workflow.md`).

### Disagreement handling

After Codex returns findings, Claude integrates them and commits — Codex
does not commit autonomously. When Codex's review and Claude's judgment
disagree:

- Low risk: Claude can override silently if the disagreement is about
  style or non-functional concerns.
- Medium risk: the rationale goes in the commit message.
- High risk: the rationale goes in a `disagreements.md` note in the
  project, or the change waits for human arbitration.

## Calling Codex on this machine

The official `codex-plugin-cc` plugin is installed at
`~/.claude/plugins/cache/openai-codex/codex/<version>/`. The runtime is
`scripts/codex-companion.mjs`. It exposes the same surface as the
`/codex:*` slash commands: `setup`, `review`, `adversarial-review`,
`task`, `status`, `result`, `cancel`.

### Two routes for invoking Codex

1. **Slash commands typed in chat by the user** — `/codex:review`,
   `/codex:adversarial-review`, `/codex:rescue`. Claude Code maps these
   onto `codex-companion.mjs review|adversarial-review|task`. This is
   the user-driven path with live streaming output.
2. **Direct invocation by Claude via Bash** — call the script directly
   when full flag control or programmatic job inspection is needed:

   ```bash
   node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs \
       review --background --base origin/main
   ```

### Pitfall to avoid: Agent + `run_in_background`

Spawning Codex via the `Agent` tool with `subagent_type:
codex:codex-rescue` and `run_in_background: true` hits a known bug
(`codex-plugin-cc` issue #198). The worktree the codex-plugin spawns is
cleaned up before Codex finishes, leaving Codex running in a deleted
directory. The Agent reports "task started in background" but nothing
usable comes back — silent-stall pattern.

Use one of:

- The slash command (user-driven, streaming output).
- Direct `codex-companion.mjs review --background` via Bash (no
  worktree race).
- `Agent` with `subagent_type: codex:codex-rescue` and
  `run_in_background: false` (foreground; blocks Claude until Codex
  finishes — fine for short reviews).

### Recovery from a stalled task

If Codex appears stuck, do NOT abandon. The codex-companion runtime
keeps job state in
`~/.claude/plugins/data/codex-openai-codex/state/<scope>/jobs/`. Three
commands recover it:

```bash
# List all jobs (running + recently finished)
node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs status --all

# Get the result of a job by ID
# (returns final output for completed; in-progress message for running)
node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs result <task-id>

# Cancel a stuck job
node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs cancel <task-id>
```

Job IDs follow the form `task-<slug>-<slug>`. They appear in
`status --all` output and in the per-job log files at the state path
above.

### Quota-window awareness

Codex CLI shares a 5-hour rate-limit window between local and
cloud-delegated tasks. If `status --all` shows multiple stuck "running"
jobs from prior sessions, cancel them first — they may hold slots that
block new dispatches from completing.

### WIP limit

Keep no more than two active Codex branches at a time. Above that,
Claude becomes a context-burning bottleneck reading verbose Codex
output instead of inspecting the actual diffs.

## Failure modes to watch

- **Superficial pass** — Codex writes mocked tests that pass without
  exercising the boundary the invariant guards. Forbid mocking critical
  boundaries; require integration tests for cross-boundary changes.
- **Concept drift** — after a rebrand or rename, Codex starts cold with
  the new terminology and quietly contradicts legacy state (e.g. legacy
  Docker volume names preserved across a rename). Always pin the
  legacy/current mapping in the brief.
- **Orchestrator context starvation** — Claude burns context formatting
  briefs and parsing verbose Codex output, leaving no budget to inspect
  the actual code. Enforce the output-discipline rule above; trim verbose
  Codex returns before reading the patch.
- **Brief drift hidden by passing tests** — if the project-context
  excerpt no longer reflects reality but the test suite happens to pass,
  Codex's confidence is unfounded. Pair every brief excerpt with a
  pointer to the canonical source so a discrepancy is detectable.

## Project responsibilities

Each project that opts into this mode must provide, in its `AGENTS.md` or
project-level instruction file:

- **Load-bearing surfaces** — explicit list of files / directories where
  the review-and-invariant gates apply.
- **Invariant tests** — at least the critical-rule subset, runnable from
  a single command.
- **Project-specific delegation deltas** — anything that diverges from
  the defaults above (additional categories Claude keeps, additional
  surfaces that need subversion probes, alternative dispatch routes).

If a project lacks an invariant-test command, the review-loop discipline
is the only enforcement and the bar for "load-bearing surface" should be
raised accordingly.
